## 5.10 系统部署方案

（本节为章节标题，子节内容在下方 5.11、5.12 中给出）

### 5.11 跨端应用架构设计

- 架构概览：后端采用 FastAPI（Python）为主，提供 REST/JSON 接口；静态资源（Tab3）由后端挂载并可通过 CDN 分发；Android 客户端为原生四 Tab 应用，Tab3 通过 WebView 加载后端静态页面；后台异步投递/批处理由独立 worker 进程或容器负责。 
- 边界与通信：
  - Android <-> 后端：HTTPS REST 调用，受 `Authorization: Bearer <token>` 保护；关键响应包含 `X-Request-Id` 以便链路追踪。
  - 后端 <-> 外部服务：通过客户端封装（GitHub、LLM、邮件），实现统一超时、重试与限流错误映射。
- 部署形态建议：分离 Web/API 与 Worker；静态资源通过 CDN/对象存储托管；后端容器化（Docker）并在 Kubernetes / Fargate 等编排下运行以支持弹性伸缩。
- 配置与运行时：全局配置通过环境变量或 Secrets 管理；支持 per-user 运行时覆盖（存储于订阅配置仓库），但敏感凭证不写入日志或公开接口。
- 可观测性：请求/异常日志、Prometheus 指标（请求延迟、外部服务错误率、队列深度）、Sentry 错误上报。

### 5.12 其他相关技术与方案

- 缓存与离线策略：采用文件快照（commit snapshots）与报告文件作为离线降级源；热点数据可在内存/Redis 缓存以减少对 GitHub 的频繁请求。
- 安全与凭证管理：生产环境使用 Vault/KeyVault/SSM 管理凭证；CI/CD 通过受控 secrets 注入构建与运行环境。
- 异步任务与幂等：邮件投递与订阅投递采用幂等 token/状态检查，避免重复发送；并发关键区由 `ConcurrencyGuard` 控制。
- 数据持久化方案：开发时使用 SQLite（本仓库实现），生产环境建议迁移到托管 RDBMS（Postgres/MySQL）并将报告快照迁移到对象存储（S3/GCS）。

## 6 数据库设计

说明：本节给出核心表结构建议，字段以项目中使用场景为准。生产中建议使用关系型数据库（Postgres），并提供迁移脚本。

1. `users`
   - id (PK, serial)
   - email (varchar, unique)
   - password_hash (varchar)
   - display_name (varchar)
   - auth_source (varchar) -- 'password' | 'github'
   - github_id (varchar, nullable)
   - github_login (varchar)
   - avatar_url (varchar)
   - is_active (boolean)
   - created_at (timestamp)
   - updated_at (timestamp)

2. `subscriptions`
   - id (PK)
   - user_id (FK -> users.id)
   - repo_url (varchar)
   - frequency (enum: instant/daily/weekly)
   - delivery_time (time)
   - last_delivered_at (timestamp)
   - enabled (boolean)
   - runtime_config (json) -- per-subscription LLM/GH overrides

3. `commit_snapshots`（或直接存为对象存储并在 DB 中保存索引）
   - id, repository (varchar), file_path (varchar), fetch_time (timestamp), metadata (json)

4. `reports`
   - id, repository, report_file_path, generated_at, generated_by_user_id (nullable), report_summary(json)

5. `job_locks` / `locks`
   - lock_key, owner, acquired_at, expire_at

6. `llm_evaluations`（可选）
   - id, request_payload (json), response (json), latency_ms, status, created_at

7. `runtime_configs`
   - id, user_id, key, value, updated_at

8. `oauth_sessions`（AuthSessionRepository 使用表，OAuth 流临时记录）
   - id, state_token, poll_token, requested_by_user_id, flow_type, redirect_uri, authorization_url,
     status, result_access_token, result_token_type, result_expires_in, result_user_json, result_repo_sync_json,
     result_message, error_code, error_message, expires_at, completed_at, created_at, updated_at

9. `weekly_data` & `project_data`（位于独立 trending DB，常量路径为 `TRENDING_DB`）
   - `weekly_data`: id, name, author, language, stars_total, forks_total, issues_total, date
   - `project_data`: id, name (unique), author, link, creation_date

说明与要点：
- `runtime_configs` 在代码中用于存储 per-user 或全局的运行时覆盖项（如 LLM/Email 配置），由 `SubscriptionRepository` 管理。
- `oauth_sessions` 保存 OAuth 登录/回调的中间态与结果（由 `AuthSessionRepository` 使用），应对 `expires_at` 与 `status` 做索引以便清理。
- `job_locks` 为分布式/进程内锁表，实现任务抢占与超时释放（由 `JobLockRepository` 使用）。
- `llm_evaluations` 用于记录 LLM 请求的元数据与性能/结果样例，利于监控与调优。
- `weekly_data` / `project_data` 存放热点仓库数据，位于单独的 `TRENDING_DB` 文件，若迁移到集中 DB，应考虑合并或同步策略。

索引与维护建议（补充）：
- 为 `oauth_sessions.expires_at`, `oauth_sessions.status` 建索引（实现已存在）。
- `job_locks.lock_key` 为主键以保证原子性；记录 `expires_at` 便于清理超时锁。
- 对 `runtime_configs` 使用 (IFNULL(user_id,0), key) 唯一索引以便覆盖与回退逻辑（代码现实现法）。
- 对 `weekly_data.date` 建索引以加速时序查询；`project_data.name` 为 unique，便于去重与替换。


索引策略：对 `users.email`, `subscriptions.user_id`, `reports.repository` 建索引；对时序检索字段（generated_at）建立时间索引用于清理和 TTL。

备份与归档：数据库每日增量备份，关键报告/快照上存对象存储并配置生命周期策略与版本化。

## 7 手机端侧部署设计

（章节 7 为顶层标题，具体内容在 7.1 中）

### 7.1 手机环境需求

- 最低支持：Android 11（API 级别 30）及以上；推荐支持到 Android 14/15（视目标用户群）。
- 构建工具：JDK 17/21，Gradle Wrapper（仓库使用 8.7），Android SDK 平台 tools/build-tools 对应版本（参见 `android-app/README.md`）。
- 设备与资源：建议设备内存 >= 2GB，存储至少预留 100MB 用于缓存和数据库（本地 `Hot_Git.db` 等）；网络：HTTPS（TLS1.2+）可用。
- 架构与 ABI：支持 armeabi-v7a 与 arm64-v8a（视 native 依赖而定）。
- 权限与安全：仅请求必要权限（网络、存储仅在需要时）；本地会话使用加密 SharedPreferences 存储（优先），否则降级为普通存储；OAuth 跳转只允许白名单域名。
- 运行时配置与调试：提供 Debug/Release 两套配置，Release 关闭 HTTP 日志与调试输出，启用 ProGuard/R8 混淆规则。

## 8 详细设计

（章节 8 为顶层标题，具体模块在 8.1、8.2 中描述）

### 8.1 交接包生成模块

#### 8.1.1 功能描述

- 功能概述：根据仓库快照与 LLM 生成规范化的“交接包”（交接包包含关键结论、风险点、变更摘要、关联 PR/commit 列表与可视化图谱链接），供页面2（晨报）与邮件投递使用。

#### 8.1.2 性能描述

- 响应时延：交接包的实时生成（非缓存）应在可接受的延迟范围内完成，目标 P95 < 8s（受 LLM 和 GitHub 响应时间影响）。
- 并发能力：模块应支持并发生成，使用 `ConcurrencyGuard` 对单仓库并发生成做串行化，避免重复计算与速率浪费。
- 资源限制：对单次生成设定超时（例如 25s）、最大重试次数（1-2 次）。

#### 8.1.3 输入

- 必要输入：`repo_url`（字符串），可选 `token`（用于 GitHub 访问），`hours`（回溯时间窗），`user_id`（用于 per-user 运行时配置）。
- 可选输入：`use_cache`（优先复用历史 report），`email_mode`（instant/manual）。

#### 8.1.4 输出

- 返回对象示例：
  - `report`（标准化 DailyReport JSON），
  - `data_file`（本地快照路径或对象存储路径），
  - `report_file`（生成的报告文件路径），
  - `used_fallback_snapshot`（bool）

#### 8.1.5 程序逻辑

1. 输入校验（repo_url 格式、hours 范围、用户鉴权）。
2. 获取 runtime 配置（若 user_id 存在，从 SubscriptionRepository 读取）。
3. 尝试从本地 cache/REPORTS_DIR 读取最近可用报告（根据 TTL）。
4. 若未命中缓存：
   - 调用 GitHub 客户端 `fetch_repo_activity(token, repo_url, hours)` 获取快照；
   - 将快照保存为 commit snapshot（JsonRepository.save_json）；
   - 调用 LLMClient.generate_report(repo_name) 生成结构化报告；
   - 标准化报告、保存到 REPORTS_DIR。 
5. 返回结果并在日志中记录 `request_id`、耗时、used_fallback_snapshot 标识。
6. 若发生外部依赖错误：
   - 若为限流/可重试错误，按指数退避进行有限重试；
   - 若最终失败，尝试回退到本地快照（如果存在），并设置 `used_fallback_snapshot=true`；
   - 抛出或返回明确错误给调用方（API 层映射为 503/429/500）。

#### 8.1.6 限制条件

- 外部依赖：GitHub API 的速率限制与 LLM 服务的可用性是主要约束，需有对应降级策略。
- 数据一致性：报告基于抓取时刻的快照，非实时全量分析，消费方需知晓报告的时间窗与来源。
- 资源与成本：频繁生成报告会产生 LLM 调用成本与 GitHub 请求开销，生产环境需合理节流与缓存。

### 8.2 交接包生成模块（扩展/集成点）

- 集成点：
  - API 层：`POST /api/project/report-by-user`、`GET /api/project/daily-report` 将调用该模块；
  - 订阅投递：SubscriptionDeliveryService 在投递时复用该模
块生成邮件正文或引用已生成的 report file；
  - 前端：页面2（晨报）、页面3（Code Panorama）通过标准 DailyReport 结构显示内容或链接到图谱页面。
- 错误处理：模块内部捕获外部错误并返回结构化错误，API 层统一映射到 HTTP 状态码并包含 `request_id`。
